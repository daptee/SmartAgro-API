1. Resumen del Sistema

SmartAgro utiliza MercadoPago Suscripciones (preapproval) para gestionar el cobro recurrente del Plan Siembra. El sistema maneja de forma automática la autorización, el cobro, los reintentos, el período de gracia y la baja del plan.

Planes

IDNombreDescripción
1Plan GratuitoPlan base sin costo. Al que se migra ante baja o impago.
2Plan SiembraPlan pago con acceso completo. Cobro mensual o anual vía MercadoPago.

Campos clave en la tabla users

is_debtor boolean
  • true → cuando un pago falla (primer aviso)
  • false → cuando paga exitosamente o cancela la suscripción
  • Se resetea: sí, en cada ciclo de pago
  • Uso: el CRON lo utiliza para recuperar suscripciones atrasadas hasta 7 días
grace_period_used boolean — NUNCA se resetea
  • true → cuando MP pausa la suscripción por primera vez
  • false → estado inicial
  • Se resetea: NO. Es permanente por diseño (una sola gracia por usuario en toda la vida)
  • Uso: determina si ante una nueva pausa se otorga gracia o se baja inmediatamente
id_plan integer
  • Refleja el plan activo del usuario. Cambia por webhooks o por cancelación manual.
  • Siempre se actualiza junto con is_debtor en los cambios de estado.

Tabla payment_history

CampoTipoDescripción
id_userintID del usuario
typestringEstado del pago: approved, rejected, paused, free_trial, etc.
dataJSONPayload completo recibido de MercadoPago
preapproval_idstringID de la suscripción en MP
payment_idbigint UNIQUEID único del pago en MP. Usado para deduplicar webhooks repetidos.
error_messagestringDetalle del error si el pago falló
Nota: El campo payment_id fue agregado en abril 2026 para resolver el problema de correos duplicados. Requiere ejecutar: ALTER TABLE payment_history ADD COLUMN payment_id BIGINT UNSIGNED NULL UNIQUE;

2. Flujo Completo de una Suscripción

2.1 Alta de suscripción

1
Usuario inicia suscripción
Frontend llama POST /subscription
→
2
API crea preapproval en MP
Estado inicial: pending
→
3
Usuario completa pago en MP
MP redirige al back_url
→
4
MP envía webhook authorized
Handler actualiza id_plan = 2
→
5
Emails de bienvenida
WelcomePlan al usuario + NotificationWelcomePlan al admin

Opción: mes gratis (free trial)

Si el plan incluye mes gratis, el preapproval se crea con free_trial.frequency = 1 month. MP no cobra el primer mes. El webhook authorized llega igual, se registra en payment_history con type = free_trial y el next_payment_date se setea 30 días después.

2.2 Pago mensual exitoso

1
MP intenta cobro en fecha de vencimiento
Webhook subscription_authorized_payment con status: scheduled
→
2
MP procesa el cobro
Webhook payment con status: approved
→
3
Sistema registra pago
PaymentHistory creado. is_debtor = false, id_plan = 2
→
4
Email al usuario
NewPayment (pago normal) o ServiceRegularized (si venía con deuda)
Deduplicación: El sistema usa el campo payment_id de MP para evitar procesar el mismo pago más de una vez, incluso si MP reenvía el webhook múltiples veces.

2.3 Pago fallido y reintentos

1
Cobro rechazado
Fondos insuficientes, tarjeta vencida, etc.
→
2
Primer aviso
is_debtor = true. Email FailedPayment enviado una sola vez.
→
3
MP reintenta automáticamente
Hasta ~4 reintentos en ~20 días. Los reintentos fallidos se loguean pero NO re-envían email.
→
4
MP pausa la suscripción
Webhook paused. Se evalúa período de gracia.
Importante: A partir del segundo reintento fallido del mismo ciclo, el sistema NO vuelve a enviar el correo de "pago fallido" al usuario. Esto evita el spam de correos que se reportó en producción.

3. Período de Gracia

Regla central: Cada usuario tiene una única gracia en toda su vida. El campo grace_period_used nunca se resetea, ni al pagar, ni al cancelar, ni al resuscribirse.

3.1 Primera vez que la suscripción se pausa (grace_period_used = false)

1
Webhook paused
MP pausó la suscripción por múltiples fallos.
→
2
Verificar suscripción real
Se verifica que el preapproval_id exista en users_plans. Evita bajar usuarios con plan asignado manualmente.
→
3
Otorgar gracia
grace_period_used = true. Plan Siembra se mantiene.
→
4
Emails de gracia
GracePeriodGranted al usuario + NotificationGracePeriodGranted al admin.

3.2 Segunda pausa o más (grace_period_used = true)

1
Webhook paused o pago fallido
El usuario ya usó su gracia.
→
2
Verificar suscripción real
Misma verificación contra users_plans.
→
3
Baja definitiva
id_plan = 1, is_debtor = false. Suscripción cancelada en MP.
→
4
Emails de expiración
ExpiredSubscription al usuario + NotificationExpiredSubscription al admin.

3.3 Tabla de decisión

grace_period_used id_plan Suscripción en users_plans Resultado
false 2 Existe ✅ Gracia otorgada — mantiene Plan Siembra
true 2 Existe ❌ Baja definitiva a Plan Gratuito
true 1 cualquiera ⏭ Duplicado ignorado — ya fue dado de baja
cualquiera 2 No existe ⚠️ Webhook ignorado — plan asignado manualmente desde admin

4. Cancelación de Suscripción

4.1 Cancelación manual por el usuario

1
Usuario cancela desde la app
Frontend llama POST /subscription/cancel con preapproval_id.
→
2
API cancela en MP
PUT a /preapproval/{id} con status: cancelled.
→
3
Baja de plan
id_plan = 1, is_debtor = false. Historial guardado.
→
4
Emails de baja
LowPlan al usuario + NotificationLowPlan al admin.

4.2 Cancelación vía webhook de MP

MP puede enviar un webhook subscription_preapproval con status: cancelled (por ejemplo si el usuario cancela desde el portal de MP directamente). El handler ejecuta la misma lógica: baja a plan 1, emails LowPlan.

5. Webhooks de MercadoPago

5.1 Tipos de webhooks recibidos

TipoEstado / AcciónQué hace el sistema
subscription_preapproval authorized Sube a Plan Siembra. Guarda en users_plans. Emails de bienvenida.
cancelled Baja a Plan Gratuito. Emails de cancelación.
paused Evalúa grace_period_used. Otorga gracia o baja definitiva.
failed Solo registra en payment_history. Sin cambio de plan.
subscription_authorized_payment scheduled (sin payment.id) Registro informativo. MP aún no cobró. Se esperan webhooks posteriores.
rejected (sin payment.id) Pago rechazado antes de generar transacción. Marca is_debtor = true. Email de primer aviso.
payment approved / authorized Marca is_debtor = false, id_plan = 2. Email NewPayment o ServiceRegularized.
payment rejected / failed Solo en primer fallo: is_debtor = true + email. Reintentos siguientes: solo log.

5.2 Protecciones anti-duplicado

MecanismoCómo funcionaCubre
Cache de idempotencia Clave mp_webhook_{id}_v{version} con TTL 24h. Si el mismo evento llega de nuevo, se ignora antes de procesar. Mismo evento reenviado por MP (timeouts, reintentos de entrega)
Deduplicación por payment_id Antes de crear PaymentHistory, se verifica que no exista un registro con el mismo payment_id. Mismo pago recibido por distintos tipos de webhook
Check is_debtor En reintentos fallidos: si ya es is_debtor = true, solo se loguea. No se re-envía email. Email duplicado por múltiples reintentos de MP en el mismo ciclo
Check id_plan != 2 Si grace_period_used = true y id_plan != 2, el webhook de baja se ignora. Webhook duplicado de baja ya ejecutada

6. CRON de Pagos

Endpoint: GET /cron-payment. Se ejecuta automáticamente 4 veces por día (02:00, 08:00, 14:00, 20:00 UTC-4).

6.1 ¿Qué suscripciones procesa?

El CRON busca en users_plans solo registros donde:

  • El usuario actualmente está en id_plan = 2 (Plan Siembra activo)
  • Caso 1: next_payment_date es hoy o mañana
  • Caso 2: next_payment_date venció en los últimos 7 días Y el usuario es is_debtor = true
  • Se toma solo el registro más reciente por usuario (unique('id_user')), evitando procesar preapprovals viejos de suscripciones anteriores fallidas

6.2 Qué hace por cada suscripción

1
Consulta estado en MP
No confía en la BD. Llama a /preapproval/{id} para obtener el estado real.
→
2
Actualiza precio
Recalcula el monto en ARS (precio USD × dólar oficial) y hace PUT en MP.
→
3
Si está pausada
Intenta reactivar (status: authorized). Si falla, evalúa gracia o baja definitiva.
→
4
Actualiza BD
Guarda la nueva data y next_payment_date de MP en users_plans.
Nota sobre el log: El campo status_bd en el log de "Suscripciones encontradas" refleja el estado guardado en la BD antes de consultar MP. El estado real de MP se logea después como "Estado actual de MercadoPago para usuario X".

8. Bugs Corregidos en Abril 2026

#ProblemaCausa raízSolución aplicada
1 Correo de baja enviado a usuario con plan asignado manualmente desde el admin El bloque paused y el de pago fallido no distinguían si el plan fue asignado por suscripción real en MP o manualmente. Se verifica que preapproval_id exista en users_plans antes de ejecutar la baja. Si no existe, el webhook se ignora.
2 Correo de baja/fallo llegó múltiples veces (hasta 3 veces con horas de diferencia) Los emails se enviaban fuera del bloque de deduplicación de PaymentHistory, y la ventana de 5 minutos no cubría los reintentos tardíos de MP. Emails movidos dentro del bloque !$existingPayment. Deduplicación cambiada a payment_id único de MP (permanente, sin ventana temporal). Reintentos normales solo loguean si is_debtor ya es true.
3 Pago rechazado sin payment.id no se procesaba MP envía el campo payment como objeto sin id cuando el rechazo ocurre antes de generar una transacción. El código retornaba error sin procesar. Se detecta el caso payment.status = rejected sin id y se procesa directamente con los datos del authorized_payment.
4 Mismo webhook procesado 4 veces en 4 segundos MP reenvía el mismo evento ante timeouts. Sin idempotencia en el handler. Cache de idempotencia por clave mp_webhook_{id}_v{version} con TTL 24h al inicio del handler.
5 CRON procesaba preapprovals viejos/fallidos del mismo usuario La query sobre users_plans (tabla de historial) devolvía múltiples registros por usuario. Agregado whereHas('user', id_plan=2) para excluir usuarios ya dados de baja, y unique('id_user') para quedarse con el preapproval más reciente.
6 CRON procesaba suscripciones canceladas de ciclos anteriores La query no filtraba por el estado actual del usuario en la tabla users. Agregado whereHas('user', fn($q) => $q->where('id_plan', 2)) a la query principal del CRON.
